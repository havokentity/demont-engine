Three backends. One C++ codebase. Mesh CSG as the headline feature, hardware ray tracing as the through-line.

This is the documentation set for a GPU path tracer being built for the M4 Max. The renderer runs interactively at 1080p with hardware ray tracing on Metal, falls back to Vulkan via MoltenVK, and has a CPU reference backend via Embree. All three are switchable at runtime.

The headline feature is interactive mesh CSG — boolean operations on imported triangle meshes via Manifold, baked on a worker thread, rendered through hardware RT. Slider drag on a leaf transform updates a drilled hole in 5–15 ms.

There is no GUI library, no embedded scripting language, no engine framework. The renderer is a server: every interaction goes through a network console (WebSocket + JSON, or plain TCP for shell scripting). The web UI is vanilla JavaScript embedded in the binary.

§ 03 Project at a glance

Headline — Mesh CSG via Manifold. Boolean operations (union, intersect, subtract) on imported triangle meshes, baked on a worker thread, rendered through hardware ray tracing. Per-face materials preserved through booleans.

Three backends, one source — Software (Embree-based CPU reference), Metal (native via metal-cpp), Vulkan (via MoltenVK on Mac, native on Windows later). Shaders written once in Slang, cross-compiled to MSL, SPIR-V, and CPU C++. Backend switching at runtime, Quake-style.

No GUI — All control flows through a network console: WebSocket on port 27960 for the rich web UI, plain TCP on 27961 for shell pipelines. Cvars and commands are the engine's only API.

Memory-aware from day zero — Tagged allocator with per-category budgets, frame arenas, pool allocators, hardware introspection at startup. mem_report tells you exactly where every byte is.

Mac-only initially — M4 Max is the target. Windows is a deferred phase, not a parallel track. The Vulkan backend is built on Mac via MoltenVK from the start; native Windows comes later as a validation pass on the abstraction.
