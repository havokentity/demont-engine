v1.0 Document 03 / 04 The How

Implementation Plan

Phase by phase, with acceptance criteria. Written for Claude Code (or another AI agent) to execute. The human user does not write CMake — the agent does.

Read the principles and design docs before starting any phase. This plan tells you the order; those docs tell you the constraints. Every phase ends with a runnable demo and a git commit.

§ 00

Ground rules for the agent

  1. Mac-first. All code must build and run on macOS (Apple Silicon, M4 Max). Windows is Phase 12. The Vulkan backend is built on Mac via MoltenVK from Phase 4.
  2. CMake is your responsibility. The user has explicitly said they struggle with CMake. Never ask the user to edit CMake files. When the build needs to change, you change it.
  3. Modern CMake (≥3.27). Targets, properties, target_link_libraries, target_include_directories. FetchContent for all third-party deps unless they need system install. No global include_directories.
  4. Generator: Ninja. Configure with cmake --preset mac-debug; build with cmake --build build/mac-debug.
  5. C++23. If something requires C++26 features, downgrade rather than require nightly compilers.
  6. No silent failures. Every shader compile, every RHI call, every Metal/Vulkan API call checks errors and logs clearly. Crash early with a useful message rather than rendering a black screen.
  7. Run the smoke test after every phase. Each phase ends with a runnable demo. Run it. Take a screenshot if relevant. Don't declare a phase done without running.
  8. One git commit per phase with the milestone tag in the message: P0: toolchain, P9: mesh CSG headline, etc.
  9. Read the principles doc constantly. When in doubt about how to structure something, the rules are there.
  10. If a phase fails: stop, write down what failed, tell the user. Don't keep iterating in a loop — get a human to look.

P0 Toolchain & memory foundation

A CMake project that builds on macOS, has the tagged memory manager wired up, and prints "hello tracer" — with mimalloc, Tracy, and global new/delete overrides already in place.

Tasks

  1. Create directory structure (empty dirs + .gitkeep). Initialize git.
  2. Write .gitignore: build/, .cache/, .idea/, .vscode/, *.metallib, *.spv, compile_commands.json, OS junk.
  3. Top-level CMakeLists.txt: cmake_minimum_required(VERSION 3.27), C++23, CMAKE_EXPORT_COMPILE_COMMANDS ON. Options: ENABLE_VULKAN_BACKEND (default OFF for now).
  4. CMakePresets.json with mac-debug and mac-release using Ninja generator.
  5. cmake/Dependencies.cmake: FetchContent for glm, fmt, mimalloc, tracy. Pin versions.
  6. core/Log: thin wrapper over fmt::print with LOG_INFO / LOG_WARN / LOG_ERROR macros.
  7. core/Memory/ module:
    • MemTag enum (Misc, Scene, MeshAssets, Textures, GpuBuffers, Csg, Console, JobSystem, Render, Stl).
    • PersistentHeap backed by mimalloc, with per-tag accounting.
    • Global operator new / operator delete overrides routing to PersistentHeap with Misc tag.
    • MemTagScope RAII for thread-local tag stack.
    • Per-tag stats accessible via GetMemReport().
  8. Tracy hooks for allocator: TracyAlloc / TracyFree in heap calls.
  9. app/main.cpp: LOG_INFO("hello tracer");, allocate something tagged, print mem report, exit.
Acceptance
  • cmake --preset mac-debug && cmake --build build/mac-debug succeeds.
  • Running the binary prints "hello tracer".
  • Memory report prints non-zero Misc tag bytes (from STL allocations etc.).
  • Tracy shows the process when connected.
  • compile_commands.json generated for clangd/CLion.

P1 Window, console, engine skeleton

A native window opens. The renderer hosts a WebSocket server. Opening http://localhost:27960 in a browser shows a console that can run quit, sys_info, mem_report. Engine class is wired with Rebuild stub. Job system runs jobs.

Tasks

  1. Add deps: glfw, civetweb (or chosen WS lib), enkiTS, tomlplusplus.
  2. app/Window: GLFW wrapper. Window(w, h, title), ShouldClose, PollEvents, GetNativeHandle() returning void*.
  3. Cocoa native handle: small .mm file using glfwGetCocoaWindow to retrieve NSWindow*. Critical: any file that includes Apple Cocoa/Metal headers must be .mm, not .cpp.
  4. app/Input: poll-based input state. Keys, mouse delta, mouse buttons. Right-click drag captures mouse.
  5. core/Memory/FrameArena: triple-buffered bump allocator. Per-frame transient.
  6. core/Memory/PoolAllocator<T>: free-list of fixed-size objects.
  7. core/Jobs/JobSystem: enkiTS wrapper. Submit, Wait, parallel-for. One worker per P-core.
  8. core/Hardware/HardwareInfo: gather at startup via sysctl, expose via cvars (sys_cpu_model, sys_cpu_pcores, sys_ram_total_mb, sys_os, etc.).
  9. console/Console:
    • CVar with name, value (string-stored), default, flags, OnChange callback.
    • Command with name, callback taking std::span<std::string_view>.
    • Static-init registration via macros: CVAR(r_max_bounces, "8", "max path tracer bounces").
    • Console::Execute(line) tokenizes, dispatches.
    • Queue from server thread → main thread for thread-safe execution.
  10. console/ConsoleServer:
    • WebSocket on port 27960 (configurable via cvar net_port).
    • Plain-line TCP on port 27961 (configurable via net_line_port).
    • Bind to 127.0.0.1 only by default.
    • Origin check on WebSocket upgrade.
    • Serves embedded index.html at /.
    • Event broadcast for log, frame_stats, mem_stats, gpu_stats topics.
  11. web/index.html + console.js + console.css: vanilla JS, no build step. Console UI with output history, input field with command/cvar tab completion via list_cvars queries, log streaming.
  12. cmake/EmbedResource.cmake: embed web/* as binary blobs in the executable.
  13. engine/Engine: top-level orchestrator. Owns Console, ConsoleServer, JobSystem, Scene (empty), Window. Tick(dt) drains command queue, updates input, renders (no-op for now). SwitchBackend(BackendType) stub that just logs "rebuild requested" — no Device yet.
  14. Initial cvars/commands wired: quit, sys_info, mem_report, echo, exec <file>, r_backend (only valid value: none).
Acceptance
  • Window opens, resizable, ESC or close button exits cleanly.
  • Browse to http://localhost:27960: console UI loads.
  • sys_info prints accurate hardware details.
  • mem_report prints per-tag breakdown.
  • quit from web console exits the app.
  • Submitting a no-op job to JobSystem and waiting on it works.
  • Tracy shows main thread + worker threads.
  • Connecting via nc localhost 27961 and typing echo hi works.
git:P1: window, network console, engine skeleton
P2 Software backend
SOFTWARE

Software backend renders a clear-color image to the window. Runtime backend switching demonstrable (Software ↔ Software, exercising the Rebuild path). Embree integrated for future ray tracing use.

Tasks

  1. Add deps: embree via FetchContent.
  2. Slang CPU spike. Before committing to Slang's CPU target, write a tiny .slang file (just output[tid] = vec4(1,0,0,1)), run slangc to generate C++, confirm it produces usable code. If not, fall back to writing the kernel directly in C++ (acceptable cost at this phase; revisit at Phase 7).
  3. rhi/: write all interface headers per design doc §4. Compile-clean, no implementations.
  4. rhi_software/SoftwareDevice:
    • Buffers as std::vector<uint8_t>, textures as aligned heap blocks.
    • Compute pipelines as function pointers (or std::function).
    • Dispatch as parallel-for via JobSystem, one tile per job.
    • Swapchain: GLFW framebuffer blit. Software writes to a CPU buffer; we use glfwSwapBuffers with GL or a CPU→texture upload via a tiny GL helper. Or simpler: use GLFW's NSOpenGLContext + glDrawPixels-equivalent path. Pick one approach and stick to it.
  5. Engine wires Software backend into Rebuild. r_backend software creates SoftwareDevice; r_backend none destroys it.
  6. First "renderer": software clear-color shader. Parameterize the color via cvar r_clear_color.
  7. Per-frame stats subscribe path wired up: emit FPS and frame-time on the frame_stats topic.
Acceptance
  • r_backend software from web console: window fills with clear color.
  • r_clear_color "0 1 0": window changes to green.
  • r_backend none followed by r_backend software: rebuild path works without leaks (mem_report shows GPU tag returning to zero between rebuilds).
  • Frame stats appear in web UI, updating live.
  • Embree links and runs (a smoke test creating an empty scene).
git:P2: software backend, runtime rebuild
P3 Metal backend
SOFTWAREMETAL

Metal backend implemented via metal-cpp. Slang compute shader clears the swapchain to a color. Runtime switch Software ↔ Metal works. The renderer code is unchanged — only the device differs.

Tasks

  1. Add metal-cpp: download Apple's package once, place in third_party/metal-cpp/. CMake creates an INTERFACE lib that links Metal/Foundation/QuartzCore frameworks.
  2. Slang setup: download slang-X.Y.Z-macos-aarch64.tar.gz, extract, cmake/Slang.cmake exposes compile_slang(TARGET FILE FORMAT).
  3. cmake/EmbedResource.cmake already done in P1; reuse for compiled shader bytecode.
  4. shaders/Clear.slang: 
    [[vk::binding(0, 0)]]
RWTexture2D<float4> output;

[shader("compute")]
[numthreads(8, 8, 1)]
void main(uint2 tid : SV_DispatchThreadID) {
    uint2 dim;
    output.GetDimensions(dim.x, dim.y);
    if (any(tid >= dim)) return;
    output[tid] = float4(1, 0, 0, 1);
}
    Compiled to MSL via slangc, embedded in binary.
  5. rhi_metal/MetalDevice: owns MTL::Device*, MTL::CommandQueue*. Resource pools for buffer/texture/pipeline handles.
  6. rhi_metal/MetalCommandBuffer: wraps MTL::CommandBuffer* + current encoder.
  7. rhi_metal/MetalSwapchain.mm: creates CAMetalLayer, attaches to NSWindow's content view via [contentView setLayer:] + [contentView setWantsLayer:YES]. Manages drawable acquisition.
  8. Same renderer code from P2 now runs on Metal. The Renderer's RenderFrame is unchanged; only the Device switches.
  9. GPU memory tracking: poll MTL::Device::currentAllocatedSize() per frame, emit on gpu_stats topic.
  10. Add r_backend metal to valid values; trigger Rebuild.
Watch out

Any file including Cocoa/Metal frameworks must be .mm, not .cpp. metal-cpp is C++ only and covers Metal but not AppKit; the layer attachment to NSWindow needs Objective-C.

Acceptance
  • r_backend metal: window clears to red via Metal compute shader.
  • Xcode GPU Frame Capture shows the compute dispatch.
  • Switch r_backend software → r_backend metal → r_backend software repeatedly: no leaks, no crashes, mem_report and gpu_stats update correctly.
  • Resize the window: swapchain recreates, renders correctly.
  • sys_gpu_name reports "Apple M4 Max".

P4 Vulkan backend (via MoltenVK)
SOFTWAREMETALVULKAN

Vulkan backend renders the same Slang clear shader on the same machine, via MoltenVK. All three backends switchable at runtime. This phase validates the abstraction is honest.

Tasks

  1. MoltenVK RT spike. Run vulkaninfo on the M4 Max. Confirm VK_KHR_ray_tracing_pipeline, VK_KHR_acceleration_structure, VK_KHR_ray_query are present. If missing or partial, we still proceed — Phase 8 (meshes) will use the fallback compute path on Vulkan instead of hardware RT extensions. Document what you found.
  2. Vulkan SDK on Mac is a hard prerequisite. CMake checks for $VULKAN_SDK, errors if missing with a clear "install Vulkan SDK from lunarg.com" message.
  3. Add deps: VulkanMemoryAllocator via FetchContent.
  4. Flip ENABLE_VULKAN_BACKEND default to ON for mac-debug.
  5. Slang now also targets SPIR-V. Update cmake/Slang.cmake to emit .spv alongside .metallib.
  6. rhi_vulkan/VulkanDevice:
    • Instance creation with validation layers ON in debug.
    • Physical device selection: pick the first one (only MoltenVK presents on Mac anyway).
    • Required extensions: VK_KHR_swapchain, VK_KHR_buffer_device_address, VK_EXT_memory_budget, plus RT extensions if available.
    • VMA for buffer/image allocation.
    • Descriptor sets matching Slang's parameter block layout.
  7. rhi_vulkan/VulkanSwapchain: VK_KHR_surface from GLFW, swapchain with explicit recreate-on-resize.
  8. Same Clear.slang renders identically.
  9. GPU mem tracking via VK_EXT_memory_budget.
  10. Add r_backend vulkan to valid values.
Performance note

Vulkan-via-MoltenVK will be slower than native Metal on the same workload, by 10-25% typically. That's the translation layer cost. It's not a bug. If you see a much larger gap, investigate, but expect some gap.

Acceptance
  • All three backends switch cleanly: r_backend software ↔ metal ↔ vulkan.
  • Each renders the same red clear via the same Slang source (compiled to different targets).
  • Vulkan validation layers report no errors.
  • gpu_stats reports correct values for both Metal and Vulkan.
  • No leaks across rebuild cycles.

P5 FPS camera & analytic primitives
SWMETALVK

Three colored spheres on a plane, lit by a directional light. FPS camera flies around. Same scene on all three backends.

Tasks

  1. renderer/Camera: position + yaw + pitch. WASD, mouse-look on right-click drag. Pitch clamped ±85°. Speed configurable, shift = sprint. Mouse sensitivity / FOV / speed exposed as cvars.
  2. Camera state changes set a "dirty" flag in Engine that flows to Renderer::ResetAccumulation (will matter from Phase 7).

    • On miss: simple sky gradient.
  5. Camera + scene uploaded as buffers per frame. Use persistent-mapped HOST_VISIBLE on Vulkan, StorageModeShared on Metal, plain memcpy on Software.
  6. Hardcoded test scene: ground plane (gray) + three spheres (red, green, blue).
Acceptance
  • Three spheres on a plane render correctly under all three backends.
  • Camera flies around with WASD + mouse-look. Right-click captures mouse.
  • Frame rate >120 FPS on Metal at 1080p, >60 FPS on Vulkan at 1080p, ~5–10 FPS on Software at internal 480×270 upscaled to 1080p (window stays 1080p regardless of backend).

P6 Materials
SWMETALVK

Three BRDFs working: Lambert, metal (mirror with optional roughness), dielectric (refraction). Visible glass refraction and metallic reflection.

Tasks

  1. shaders/Material.slang: tagged-union Material struct (type, albedo, roughness, ior, emission). SampleMaterial function returning new direction + throughput.
  2. Lambert: cosine-weighted hemisphere.
  3. Metal: GGX importance sampling (Walter et al. 2007). roughness=0 → mirror.
  4. Dielectric: Snell's law + Schlick Fresnel approximation. Handle total internal reflection.
  5. Path tracing loop: bounce up to 8, multiply throughput, Russian-roulette terminate after depth 4.
  6. PCG RNG seeded per-pixel per-frame from (pixel.x, pixel.y, frame_index).
  7. Test scene: ground (Lambert grey) + three spheres: red Lambert, gold metal, glass dielectric (IOR 1.5).
Acceptance
  • Glass sphere shows recognizable refraction with TIR at grazing angles.
  • Metal sphere shows mirror reflection of red sphere and ground.
  • No NaNs (verify with debug visualization toggle: r_show_normals, r_show_albedo).
  • All three backends produce visually identical output for the same scene + RNG seed.

P7 Render modes & path tracing with accumulation
SWMETALVK

Three render modes online: preview (primary + direct light, cheapest), raytrace (Whitted-style real-time reflections + refractions), pathtrace (Monte Carlo with accumulation, headline). Resolution decoupled from window size — internal res scaled by r_resolution_scale, upscaled to fill 1080p window with aspect preserved.

Tasks

  1. Resolution decoupling. Renderer takes (window_w, window_h, scale). Internal HDR target sized at round(window × scale) for both dimensions — aspect always preserved. Tonemap pass upscales internal → swapchain via bilinear sample. Window stays at 1080p (or whatever app_window_* say) regardless of scale or backend.
  2. Render mode switch. r_mode cvar with values preview | raytrace | pathtrace. Single Slang shader, three inner loops gated by a kRenderMode push constant. Switching modes resets accumulation.
  3. Preview mode. Cast primary ray, take closest hit, evaluate one direct-light source (the existing directional from P5/P6), write radiance, done. No recursion, no accumulation buffer needed (writes directly to swapchain via tonemap with sample_count=1).
  4. Raytrace mode. Whitted-style. Primary ray + recursive reflection/refraction up to r_raytrace_depth (default 4). Materials: dielectric splits into reflect+refract weighted by Fresnel; metal does perfect mirror reflection; Lambert terminates with direct light. Hard shadows via shadow ray to the directional source. Real-time on hardware RT, no Monte Carlo, no noise, no accumulation.
  5. Pathtrace mode. The full Monte Carlo path tracer from earlier draft. HDR accumulation buffer (RGBA32F at internal resolution). Path accumulates radiance / N samples per pixel; N = sample_count. Reset on camera movement, mode change, or scene change.
  6. Tonemap pass reads accumulation buffer (or preview/raytrace's direct output), applies ACES, gamma-corrects, upscales to swapchain.
  7. Cornell box test scene wired up via hardcoded primitives or first cut of TOML scene loading.
  8. Per-frame stats include mode, spp (in pathtrace mode), internal_resolution in frame_stats.
Acceptance
  • Window is 1080p on all three backends. No backend ever resizes the window or causes black bars.
  • r_mode preview renders a Cornell box scene at >240 FPS on Metal, >5 FPS on Software at r_resolution_scale 0.25.
  • r_mode raytrace shows mirror sphere reflecting the room and a glass sphere refracting it, real-time at 1080p on Metal (>120 FPS).
  • r_mode pathtrace converges to a clean image in <5 seconds when standing still on Metal.
  • Camera movement immediately resets pathtrace accumulation; image is noisy at 1 spp during motion.
  • r_resolution_scale 0.13 on the software backend: internal render is ~256×144 but the window is still 1920×1080 with the image upscaled cleanly. No distortion, no letterbox.
  • Pathtrace image identical across backends after enough samples (modulo RNG ordering).

P8 Triangle meshes via hardware ray tracing
SWMETALVK

glTF mesh loads, builds BLAS/TLAS on Metal and Vulkan, builds Embree BVH on Software. Renders correctly via hardware RT (Metal/MoltenVK) or Embree (Software).

Tasks

  1. Add deps: tinygltf, stb_image.
  2. Mesh loader returns positions, normals, indices, per-face material IDs as SoA arrays.
  3. Extend RHI: CreateBLAS, CreateTLAS, command buffer methods to build them.
  4. Metal backend: MTL::PrimitiveAccelerationStructureDescriptor, InstanceAccelerationStructureDescriptor. Build via MTL::AccelerationStructureCommandEncoder.
  5. Vulkan backend: vkCmdBuildAccelerationStructuresKHR if available; otherwise log "MoltenVK lacks RT extensions, falling back to compute-shader trace" and use a CPU-side BVH uploaded as buffer + traversed in the compute kernel.
  6. Software backend: build Embree RTCScene, traverse via rtcIntersect1. Embree's BVH is best-in-class CPU-side.
  7. Update path tracer kernel to use RaytracingAccelerationStructure (Slang). Inline ray queries for now. Index by primitiveID + instanceID for material lookup.
  8. Test: load Stanford bunny or Suzanne (glTF), render alongside analytic primitives. Verify hardware RT is used (Xcode GPU capture shows intersector calls on Metal).
Acceptance
  • Imported mesh renders correctly on all three backends.
  • Performance on Metal is dramatically faster than equivalent analytic-only scene with same triangle count.
  • Mesh can be moved/rotated by transforming TLAS instance.
  • Per-face material lookup works.

P9 ★ Mesh CSG via Manifold — headline
HEADLINE

The feature this whole project exists for. Subtract a sphere mesh from a box mesh. Edits feel real-time. Per-face materials preserved through booleans. 3-deep nested CSG trees render correctly.

Tasks

  1. Add manifold via FetchContent. It builds with CMake — straightforward.
  2. renderer/Csg/CsgNode: tree on the heap. Leaf references a MeshAsset. Inner nodes have op + two children + transform.
  3. renderer/Csg/Evaluator:
    • Walks tree, recomputing dirty subtrees with Manifold.
    • Leaves: convert MeshAsset to manifold::Manifold, cache.
    • Inner nodes: apply transforms, then + (union), ^ (intersect), - (subtract).
    • Returns BakedMesh with positions/normals/indices/face_material_ids.
  4. Per-face material: Manifold's MeshGL exposes originalID per triangle. Build originalID → material_id side table at leaf-conversion time.
  5. Critical: run Manifold evaluation on a worker thread. Use JobSystem. Don't block the render loop on a 50ms worst-case bake.
  6. On baked mesh update: upload new vertex/index buffers (allocate larger if needed), full BLAS rebuild, update TLAS instance, reset accumulation. All on the next frame.
  7. Console commands for CSG editing: 
    csg_create_leaf <node_id> <mesh_asset> <material_id>
csg_create_op   <node_id> <union|intersect|subtract> <left_id> <right_id>
csg_set_transform <node_id> <tx> <ty> <tz> <rx> <ry> <rz> <scale>
csg_set_op <node_id> <new_op>
csg_delete <node_id>
csg_set_root <node_id>
csg_dump  # prints the tree
  8. Initial test scene from console: (box - sphere), the "drilled cube" classic. Then (box - sphere1 - sphere2 - sphere3). Then a 3-deep nested tree.
  9. Web UI gets a basic CSG tree view (vanilla JS) with sliders for leaf transforms. Slider drag → command stream → live re-bake.
Manifold prerequisite

Operand meshes must be manifold and watertight. Test assets should be primitives (box, sphere, cylinder) generated in code or carefully chosen glTF files. Most internet-sourced models aren't manifold and will be rejected — that's not a bug.

Acceptance
  • Slider drag on a leaf transform: hole position moves at >10 Hz update rate.
  • Per-face materials work: metal sphere subtracted from glass box → hole interior is metal.
  • 3-deep nested CSG tree renders correctly: ((box - sphere1) ∪ cylinder) - sphere2.
  • No frame hitches during edits — Manifold runs on worker thread.
  • Accumulation resets only when baked mesh changes, not every frame.
  • All three backends render the result identically.

Notes the agent should respect

  • Never put Apple framework headers in .cpp files. Use metal-cpp (C++ only) or rename to .mm (Objective-C++).
  • Slang's [[vk::binding(...)]] annotations double as Metal binding hints. Use them consistently.
  • Texture format conversions live in a centralized table. Don't scatter them.
  • Coordinate conventions: glm right-handed Y-up Z-out. Don't introduce handedness flips.
  • NDC differences between Metal and Vulkan don't matter for compute-only ray tracing. Revisit only if we add rasterization.
  • When a phase fails: stop, document, ask the user. Don't loop.
