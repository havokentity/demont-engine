Architecture Principles

The rules of the road. Every other document cites this one. When in doubt, this document wins.

This is a constraints document, not a tutorial. It tells you what the project commits to and — equally important — what it explicitly rejects. The rejections matter, because they prevent the architecture from drifting toward whatever pattern is locally convenient.

Rules are tagged HARD (do not violate, ever) or SOFT (default — exceptions need a comment explaining why). Rules without tags are descriptive context, not enforcement.

§ 01

Project identity

This is a real-time GPU path tracer with mesh CSG as the headline feature. Three backends — Software, Metal, Vulkan-via-MoltenVK — runtime-switchable. macOS / Apple Silicon (M4 Max) is the only target platform until further notice; Windows is a deferred phase, not a parallel track.

It is not a game engine. Not a scene editor. Not a shader playground. Not a vehicle for learning every graphics concept. Features that don't serve the path tracer — audio, physics, networking other than the console, embedded scripting, asset pipelines beyond glTF/OBJ load — stay out.

The simplest project that ships the headline feature is the right project. Every additional system has to earn its place against the cost of carrying it.
§ 02

Performance discipline

"Hardcore performance" means optimizing what matters and leaving alone what doesn't. The per-frame hot path and the GPU shader code are the things that matter. Startup, asset loading, console parsing, scene save/load — they don't, and we do not spend cycles on them.

  • HARD Performance work is profile-driven. No optimization lands without a Tracy trace or a measurement showing it helps. "I think this might be faster" is not justification.
  • HARD Premature SIMD intrinsics are forbidden. We do not write _mm_*, vld1q_*, or any architecture-specific intrinsic until the profiler points at a specific loop and compiler auto-vectorization has been tried and proven insufficient.
  • HARD Allocations in the per-frame hot path are forbidden outside the frame arena. Heap allocations during a frame are a build-error-tier bug.
  • SOFT Prefer libraries written by people who know more than us about their domain: Embree for CPU ray tracing, mimalloc for general allocation, glm for vector math, Manifold for mesh booleans, highway for portable SIMD when we eventually need it.
Optimize where the profiler points, never where instinct does. The hardcore flex is using the right libraries, not writing assembly.

What "fast" means here

Targets, not promises:

WorkloadTarget on M4 Max
Path tracer (pathtrace), 1080p, 1 spp, no denoiser≥ 60 FPS on Metal
Path tracer (pathtrace), 1080p, 1 spp, with denoiser≥ 30 FPS on Metal
Whitted raytrace (raytrace), 1080p≥ 120 FPS on Metal — real-time and clean
Manifold CSG re-bake, moderate operands< 15 ms

§ 03

Memory management

The project commits to a real memory manager from Phase 0. This is non-negotiable — retrofitting allocators after the fact is the kind of work that consumes weeks for nothing.

Allocator categories

Different lifetimes want different allocators. We do not try to write one allocator that's good at everything.

AllocatorLifetimeCostUsed for
Frame arenaOne frameAtomic addPer-frame transient: command staging, debug strings, upload prep
PoolObject-by-objectO(1) free-listSame-size objects, frequent alloc/free: jobs, BVH nodes, log entries
Persistent heapLong-livedMimallocScene data, asset storage, anything not transient
GPU heapResource-by-resourceSub-allocationWraps Metal MTLHeap / Vulkan VMA. One MTLBuffer/VkBuffer per heap, sub-allocated

Tagging is mandatory

Every allocation through our system is tagged with a MemTag at the call site. Tags are something like Scene, MeshAssets, Textures, GpuBuffers, Csg, Console, JobSystem, Render, Misc. The allocator records: total bytes per tag, peak bytes per tag, allocation count per tag.

{
    MemTagScope scope(MemTag::Csg);
    auto* tree = heap.New<CsgTree>(...);  // tagged Csg
    // any allocation made inside this scope inherits the tag
}
  • HARD Global operator new and operator delete are overridden in our code path to route through the persistent heap with a Misc tag, with a warning logged in debug. Untracked allocation does not exist in our code.
  • HARD Each tag has a configurable hard budget (cvar mem_budget_<tag>_mb). Exceeding the budget aborts in debug, returns null in release, always logs.
  • SOFT STL containers default to system allocator and that's fine for now. If a specific container shows up in mem_report as significant and untagged, we either replace it with a custom container that takes an explicit allocator, or we accept the imprecision.
  • SOFT Third-party libraries use whatever allocator they want. Our visibility is into our code. Library allocations are opaque, which is fine.
Every byte we allocate has a name and a budget. Memory we cannot see is memory we cannot reason about.
§ 04

Parallelism

Parallel from Phase 1, not retrofitted later. A real job system from day one — enkiTS or marl are good choices, hand-rolled is also fine if it stays under ~400 lines.

  • HARD The render loop is single-threaded. GPU command submission is sequential. We do not multi-thread render submission. Everything feeding the render is parallel.
  • HARD All CPU work that takes longer than ~1 ms is expressed as a job: CSG evaluation, BVH build, scene upload preparation, asset loading, screenshot encoding.
  • HARD Thread communication is via lock-free SPSC ring buffers. No std::mutex in the per-frame path. Mutex-guarded structures live in setup code only.
  • SOFT One worker per performance core. The M4 Max has 12 P-cores; the efficiency cores stay idle by default. Don't oversubscribe.
  • SOFT Use alignas(64) on contended atomics and shared cache lines to avoid false sharing.
§ 05

Data layout

Data-oriented where it matters, pragmatic where it doesn't. We are not building a pure ECS; we're building a renderer that respects cache locality in the places it shows up in the profiler.

Where DOTS-style design wins here

  • Scene as struct-of-arrays. Meshes, materials, CSG nodes, instance transforms each in their own flat arrays. Uploads to GPU buffers without marshaling.
  • Hot/cold splitting. Per-mesh "always touched per frame" data (transform, AABB, BLAS handle) in one struct; "only touched on edit" data (source path, debug name, original CPU vertex data) in a separate structure indexed by the same ID.
  • Integer IDs, not pointers. Strong-typed IDs (MeshId, MaterialId as distinct types) → zero runtime cost, big readability win, prevents whole bug class.

Where it doesn't

  • No ECS framework. We have dozens-to-low-hundreds of scene entities. Frameworks like entt only earn their cost at tens of thousands. Typed arenas + integer IDs do the same job at our scale.
  • Engine, Console, Window, Camera are plain classes. Forcing them into ECS components is busywork.
  • HARD Scene data is SoA. Multiple flat std::vector<T> arrays per type, indexed by typed ID. Not array-of-structs.
  • SOFT Strong types for IDs: struct MeshId { uint32_t v; }; not uint32_t.
§ 06

Virtual dispatch policy

OOP is not the enemy. Pointer-chasing inheritance hierarchies and unnecessary indirection are. The rule is frequency-aware: virtual dispatch is fine when the call rate is low; data layout matters when the call rate is high.

Virtual is fine

  • RHI Device, CommandBuffer — called O(100) per frame
  • Material SampleBSDF when only a handful of types — but on the GPU side this becomes a tagged-union switch in the shader anyway
  • Console Command dispatch — called from human input

Virtual is wrong

  • Anything called per ray, per pixel, per vertex
  • Inner loops of acceleration structure traversal
  • Per-allocation hooks (use templates / CRTP / direct dispatch)
§ 07

The three-backend rule

The project supports three backends — Software, Metal, Vulkan-via-MoltenVK — and they all run on the same M4 Max for now. Windows comes online much later as a Vulkan target.

  • HARD Every renderer feature lands on all three backends in the same phase. We do not ship a feature on Metal first and "port to Vulkan later." That promise has killed every cross-platform engine project that's tried it.
  • HARD No backend-specific code outside src/rhi_metal/, src/rhi_vulkan/, src/rhi_software/. The renderer never includes a Metal or Vulkan header.
  • HARD Shaders are written once in Slang and cross-compiled. We do not maintain parallel MSL/GLSL/HLSL sources.
  • SOFT The Software backend is a correctness reference, not a perf target. It is allowed to be 100× slower than the GPU backends. Its job is to catch bugs the GPU is hiding.
An abstraction is only as good as the number of concrete things it abstracts over. Three is enough to keep us honest; one is not.

The Engine/Rebuild model

Backend switching at runtime requires that GPU resources be derivable from a CPU-side scene description. The CPU scene is the source of truth; the GPU state is built from it on demand. Switching backends is: drain the old device → destroy the renderer → destroy the old device → create the new device → create the renderer → upload scene → continue. Quake-style vid_restart. Brief pause, no zero-cost hot swap promise.

§ 08

Code style and dependencies

Things this project does not include

  • HARD No GUI library. No Dear ImGui, no Qt, no anything. Tooling lives in the network console.
  • HARD No embedded scripting language. No Lua, no Python via pybind11, no AngelScript. If we need scripting, the WebSocket console allows external scripts (in any language) to drive the renderer.
  • HARD No engine framework. No Unreal, no Unity, no O3DE, no Filament. Everything is hand-built on the RHI.
  • SOFT No Boost. No Eigen. No QtCore types. We use the standard library, glm, and a small set of carefully chosen libraries listed in the design doc.

Things this project does use

The full dependency list lives in the design doc. Briefly: glm for math, GLFW for windowing, mimalloc for allocation, Tracy for profiling, Slang for shaders, metal-cpp for Metal bindings, Vulkan SDK + VMA for Vulkan, Embree for software ray tracing, Manifold for mesh CSG, tinygltf for asset loading, toml++ for scene files, civetweb (or similar single-header WS server) for the network console, fmt for logging.

C++ dialect: orthodox C++23

"C++" is a family of dialects. Pick one and stick to it. The dialect for this project is sometimes called Orthodox C++ — modern features used where they pay rent, pre-C++11 baroque excess kept out. Think id Software's later codebases, AAA game engines, the demoscene's pragmatic C++ — not Sean Parent's GoingNative talks.

The C++ Cheatsheet walks through the modern features with examples. The lists below are the enforcement specification.

Use freely

  • SOFTC++23 baseline. std::expected, std::print, deducing this where useful.
  • SOFTRAII for ownership. std::unique_ptr as the default. std::shared_ptr only when ownership is genuinely shared (rare). Raw owning pointers forbidden.
  • SOFTStrong types via single-member structs. struct MeshId { uint32_t v; }; not using MeshId = uint32_t;.
  • SOFTstd::expected<T, E> for fallible returns. Replaces both bool-out-pointer and exception throwing.
  • SOFTstd::span and std::string_view for non-owning views. Replace pointer + length pairs.
  • SOFTDesignated initializers (C++20) for any struct with more than two fields.
  • SOFTenum class always. Plain enum never.
  • SOFTRange-based for, auto when the type is obvious or doesn't matter, const auto& for read-only iteration.
  • SOFTconstexpr wherever it's free.
  • SOFTfmt::print / std::print for output. No iostream.
  • SOFTLambdas for one-shot local callbacks. Capture explicitly ([&x, y]) when reasonable; [&] is fine for short bodies.

Use carefully (with a comment justifying)

  • SOFTTemplates for generic data structures (PoolAllocator<T>, RingBuffer<T>). One template parameter, one obvious purpose. No template metaprogramming dressed up as a generic container.
  • SOFTSTL containers (std::vector, std::string, std::array) in cold code (config parsing, asset loading, console). Not in the per-frame hot path.
  • SOFTOperator overloading only for math types (vector add, matrix multiply). Never for control flow or "clever" DSLs.
  • SOFTMethod functions on structs when there's clear cohesion (camera.move(dir, dt)). Free functions are also fine — pick one per type and stay consistent.
  • SOFTif constexpr for compile-time branching. Replaces 80% of cases that needed SFINAE in older C++.

Forbidden

  • HARDExceptions. Build with -fno-exceptions. Errors via std::expected or explicit return codes. Wrap third-party libs that throw at the boundary.
  • HARDRTTI. Build with -fno-rtti. No dynamic_cast, no typeid.
  • HARDMultiple inheritance. No exceptions.
  • HARDClass hierarchies more than 2 levels deep. RHI's Device with three concrete implementations is the entire inheritance budget.
  • HARDTemplate metaprogramming. No SFINAE, no enable_if chains, no constexpr-if recursion, no type lists, no compile-time dispatch tables.
  • HARDstd::iostream. Slow, ugly syntax, incomprehensible errors. Use fmt.
  • HARDstd::regex. Notoriously slow and bloated. Hand-write parsers for the small grammars we have.
  • HARDstd::function in hot paths. Heap allocates, type-erases, can't inline. Fine for command callbacks (called from human input). Forbidden inside the renderer per-frame loop.
  • HARDSTL algorithms with cute lambdas in hot paths. std::sort is fine. std::transform_reduce with inline lambdas in a per-frame loop is harder to read and harder to optimize than a plain for loop.
  • HARDC++ modules. Compiler and build-system support is still patchy in 2026, gains for a project this size are marginal. Headers and .cpp files.
  • HARDCoroutines (co_await, etc.). Excellent for async I/O — we don't have async I/O. Job system handles parallelism.
  • HARDConcepts as a primary tool. requires clauses are cool but rarely earn their cost. If you find yourself writing one, you're probably overengineering the template.
  • HARDBoost. Eigen. Qt. Anything that brings its own world.

Smart pointer policy

Smart pointers are not interchangeable. The defaults matter.

  • SOFTstd::unique_ptr<T> is the default for any heap-allocated object. Zero runtime cost (same size as a raw pointer, no atomic ops), eliminates the entire class of "who deletes this" bugs, makes ownership visible in the type. Construct via std::make_unique<T>(args...).
  • HARDnew never appears in our code outside of make_unique's implementation. Seeing new Foo(...) in a code review is a defect.
  • HARDstd::shared_ptr<T> requires written justification at the declaration site. Reference counting has real costs (atomic increment/decrement on every copy, separate control block allocation, cycle leak risk) and "multiple things might own this" is usually a sign that ownership wasn't thought through. Default is no shared_ptr in this project.
  • HARDstd::weak_ptr<T> requires written justification, same reasoning. Existence of a weak_ptr usually indicates a shared_ptr cycle that should have been a unique_ptr + raw pointer.
  • SOFTRaw pointers are fine for non-owning references. Modern C++ didn't kill raw pointers — it killed raw owning pointers. T* means "I refer to this, someone else owns it and keeps it alive longer than me." Use T& when null doesn't make sense.
  • SOFTInteger handles for opaque resources. RHI handles (BufferHandle, TextureHandle) are integer IDs, not pointers. Better debugability, clean lifetime, usable across backend boundaries. See design §4.

See the cheatsheet §3 for the full discussion of why the defaults are what they are.

Build flags this implies

# Common
-std=c++23 -fno-exceptions -fno-rtti
-Wall -Wextra -Wpedantic -Werror=return-type -Wshadow

# Debug
-O0 -g -fsanitize=address,undefined

# Release
-O3 -ffast-math -DNDEBUG

Note on -ffast-math: it allows the compiler to assume no NaNs/Infs, which can matter in path tracer numerical paths. Revisit if it bites; per-translation-unit override is fine for shader-adjacent C++.

Modern C++, used as a better C with strong types and RAII. Boring on purpose.
§ 09

Observability

You cannot fix what you cannot see. Observability is foundational, not optional, and it's wired up in Phase 1 alongside the console.

  • HARD Tracy profiler integrated from Phase 1. Every job, every render phase, every allocation hook instrumented.
  • HARD Hardware introspection at startup: CPU model, core counts, RAM, GPU, VRAM, OS version. Exposed as read-only cvars (sys_*).
  • HARD Per-frame stats streamed to the network console: FPS, frame time breakdown, sample count, current backend, per-tag memory usage, GPU memory usage.
  • SOFT Every backend exposes currentAllocatedSize()-equivalent for GPU memory: Metal native, Vulkan via VK_EXT_memory_budget.
Console as universal control surface

Cvars and commands are the engine's API to the outside world. Every internal state worth tweaking gets a cvar. Every action worth triggering gets a command. The web UI, CLI tools, and external scripts all share this surface. See document 04.

§ 10

Scope discipline

The single biggest threat to this project is scope creep — adding "while we're at it" features that compound into a never-shipping engine. The phased plan exists to fight that. Each phase has acceptance criteria; we don't move on until they pass; we don't pull features forward from later phases just because they sound interesting.

  • HARD One milestone at a time. No starting Phase N+1 work while Phase N is incomplete.
  • HARD Acceptance criteria for a phase are the only bar. "Quality" improvements that don't move the criteria forward are feature creep.
  • SOFT When tempted to add something not in the plan, write it down in a "future" list and keep going. Revisit at the end of the current phase.
Shipping the headline feature on one platform is a better outcome than a half-built engine on three.

That said: the architecture is designed so that extending — adding a fourth backend, adding new material models, adding new geometry types — is a localized change, not a rewrite. Discipline now buys flexibility later.
