v1.0 Document 02 / 04 The Why

Design Document

A real-time GPU path tracer with mesh CSG as the headline feature. Three backends, runtime switchable, on a single C++ codebase. macOS / Apple Silicon for now.

This document explains what we're building and why the architecture is shaped the way it is. The implementation plan tells you the order of operations; the principles doc tells you the constraints. Read those after this one.

§ 01

Goals & non-goals

Goals

  • GPU-accelerated path tracer running interactively (≥30 FPS at 1080p with denoising) on M4 Max via Metal.
  • Three backends — Software (CPU reference via Embree), Metal (native), Vulkan (via MoltenVK on Mac, native on Windows later) — runtime-switchable mid-app.
  • Single C++ codebase, single shader source via Slang cross-compilation.
  • Three render modes: pathtrace (Monte Carlo with accumulation, headline quality), raytrace (Whitted-style real-time reflections + refractions + hard shadows), preview (primary rays + direct light, fastest). Switchable at runtime via cvar.
  • Hardware ray tracing on Metal and (when available) Vulkan; Embree-based BVH on Software.
  • First-class mesh CSG: union, intersect, subtract on imported triangle meshes, baked via Manifold, edits feel real-time.
  • Standard physically-based materials: Lambert, conductor, dielectric. Per-face material preservation through CSG.
  • FPS camera, free-fly controls.
  • Progressive accumulation when camera is still in pathtrace mode; 1-spp + denoiser when moving (later phase).
  • Resolution decoupled from window size. The window is always at the configured size (default 1080p). Internal render resolution is window × r_resolution_scale, upscaled to fill the window with aspect ratio preserved. A 256×144 internal render fills a 1920×1080 window cleanly — no letterboxing, no distortion. Lets the slow software backend stay usable without shrinking the window.
  • Network-only console for control: WebSocket + JSON, vanilla JS frontend embedded in binary.
  • Tagged memory manager with per-category budgets; full hardware introspection.

Non-goals (initial)

  • Scene graph or visual editor beyond the network console UI.
  • Audio, physics, networking-for-multiplayer, embedded scripting language.
  • Linux / mobile / web (Linux comes free with Vulkan but isn't a target).
  • Animated booleans (boolean ops driven by per-frame animation rather than user edits) — possible later via interval CSG, not in v1.
  • Cross-system CSG (subtracting an SDF from a mesh body or vice versa).
  • SDF objects. Discussed in §9 as future work. Hardware ray tracing accelerates triangle BVHs only — SDFs are software sphere-tracing, which throws away the M4 Max's biggest hardware advantage. Out of v1 scope. The architecture is designed to accommodate them later as a parallel geometry path.
  • Native Windows build until Phase 12 — Mac-only for now.
§ 02

Architecture overview

Four layers. The App owns the window and main loop. The Engine orchestrates the renderer, scene, and console; it owns the current Device and can swap it. The Renderer contains the path tracer, scene representation, and frame logic — it knows what to render but never how the GPU is commanded. The RHI is the abstraction line; below it live the three backend implementations.

App · Window · Input · Main loop
↓
Engine · Scene · Console · Backend orchestration
↓
Renderer · Path tracer · Camera · Frame graph
↓
RHI · Device · CommandBuffer · Resources · AccelStruct
↓

Metal
metal-cpp
Vulkan
via MoltenVK

Shaders flow through a parallel pipeline: .slang sources → slangc → MSL bytecode (Metal), SPIR-V (Vulkan), or generated C++ (Software). The Renderer doesn't know which target it's loading; the RHI does.

§ 03

Engine & runtime switching

The Engine class is the top-level orchestrator. It owns: the current Device, the Renderer, the Scene (CPU-side), the Console, the JobSystem. The App constructs an Engine and ticks it once per frame.

Critically: the Renderer does not own its Device. The Engine does. This separation enables Quake-style runtime backend switching (vid_restart): drain the old device, destroy the renderer, destroy the device, create a new device against the same window's native handle, recreate the renderer, upload scene, continue. Brief pause (~0.5–2 seconds), no zero-cost hot swap promise.

class Engine {
public:
    void Tick(float dt);                    // called every frame by App
    void SwitchBackend(BackendType target); // triggers Rebuild
    Console& GetConsole();
    Scene& GetScene();

private:
    void Rebuild(BackendType target);       // teardown + recreate

    NativeWindowHandle window_;
    std::unique_ptr<rhi::Device>   device_;
    std::unique_ptr<Renderer>      renderer_;
    std::unique_ptr<Scene>         scene_;        // CPU-side, survives switch
    std::unique_ptr<Console>       console_;
    std::unique_ptr<ConsoleServer> console_server_;
    std::unique_ptr<JobSystem>     jobs_;
};

The Scene survives backend switches because it's CPU-side data. The Renderer is reconstructed from the Scene on each Rebuild. GPU resources (BLAS, TLAS, vertex buffers, textures, accumulation buffer) are derived state — destroyed with the old device, re-uploaded into the new one.

This is also the right architecture for...

Hot reload of shaders, scene file reload, GPU device reset recovery, swapchain recreation on resize, debug feature toggles ("disable hardware RT, fall back to compute"). Backend switching is just one of many uses of the Rebuild path.

§ 04

RHI interface

The RHI ("Render Hardware Interface") is a thin C++ abstraction over the three backends. Design principle: lowest common denominator of Metal, Vulkan, and CPU compute, with explicit escape hatches. We don't try to hide everything — acceleration structures, for example, are exposed directly because they're a real concept on all three.

Handles

All RHI resources are referenced by 64-bit opaque handles, not raw backend pointers. Cheap to copy, debuggable, decouples lifetime from C++ object identity. The backend keeps a slot allocator internally.

struct BufferHandle      { uint64_t id = 0; };
struct TextureHandle     { uint64_t id = 0; };
struct PipelineHandle    { uint64_t id = 0; };
struct AccelStructHandle { uint64_t id = 0; };
struct ShaderHandle      { uint64_t id = 0; };

Device

The factory entry point. Creates resources, pipelines, acceleration structures. Acquires command buffers. Manages the swapchain.

class Device {
public:
    static std::unique_ptr<Device> Create(BackendType, NativeWindowHandle);

    BufferHandle    CreateBuffer(const BufferDesc&);
    TextureHandle   CreateTexture(const TextureDesc&);
    PipelineHandle  CreateComputePipeline(const ComputePipelineDesc&);
    AccelStructHandle CreateBLAS(const BLASDesc&);
    AccelStructHandle CreateTLAS(const TLASDesc&);

    CommandBuffer*  AcquireCommandBuffer();
    void            Submit(CommandBuffer*);
    void            WaitIdle();

    bool SupportsHardwareRT() const;
    const char* GetDeviceName() const;
    size_t GetCurrentAllocatedBytes() const;  // for memory introspection
};

Command buffer

Records work to be submitted to the GPU. The same shape works for compute dispatches and ray tracing dispatches; both are expressed as compute kernels with optional acceleration structure binding.

class CommandBuffer {
public:
    void BindComputePipeline(PipelineHandle);
    void BindBuffer(uint32_t slot, BufferHandle, size_t offset = 0);
    void BindStorageTexture(uint32_t slot, TextureHandle);
    void BindAccelStruct(uint32_t slot, AccelStructHandle);
    void PushConstants(const void* data, size_t size);

    void Dispatch(uint32_t gx, uint32_t gy, uint32_t gz);

    void CopyBufferToTexture(BufferHandle, TextureHandle);
    void Barrier(const BarrierDesc&);  // explicit; we don't auto-track
};

Why this shape

  • No render passes. We're a compute / ray-tracing renderer — no rasterization in v1. Add later only if needed for hybrid rendering.
  • Flat binding table rather than emulating Vulkan descriptor sets or Metal argument buffers. Slang controls layout; binding slots line up across backends naturally.
  • Explicit barriers. Metal and Vulkan both want them. Auto-tracking is a tarpit.
  • Acceleration structures are first-class. Hiding them costs more than it saves.

What we deliberately don't abstract

Multi-queue / async compute. Sparse resources. Mesh shaders. Bindless resources. Add when proven necessary, not before.

§ 05

Shader stack: Slang

Slang is the load-bearing decision in this architecture. One .slang source compiles to MSL (Metal), SPIR-V (Vulkan), and (via Slang's CPU target) a C++ function for the Software backend. Without it we'd write every shader three times.

  • Modules and import — split BRDFs, sampling, intersection, CSG ops into separate .slang files. No giant single-file shader.
  • Generics — interface-based material dispatch without per-ray switches.
  • ParameterBlock<T> — matches the RHI's flat binding table cleanly.
  • RayQuery / TraceRayInline — Slang's inline ray query intrinsics map to Metal's intersector and Vulkan's rayQueryEXT. Same shader code, both backends.

Compilation strategy: build-time for release (CMake invokes slangc, embeds bytecode as binary blobs in the executable), runtime in debug for hot-reload during development (link the Slang runtime, recompile on save).

§ 06

Renderer architecture

The Renderer is a sequence of compute kernel dispatches per frame. The kernel itself has three modes — chosen by cvar — that share intersection and material logic but differ in their inner loop. Tone mapping and presentation are common to all three.

The three render modes

ModeInner loopCost / pixelQuality
previewPrimary ray, hit, direct light from one source, write radiance, done.1 rayDiffuse + specular shading. No reflections. Hard shadows optional. Looks like a flat-shaded modeler.
raytracePrimary ray + recursive reflection/refraction up to r_raytrace_depth. Deterministic — no Monte Carlo, no noise.2–8 rays typicallyWhitted-style. Mirror reflections, glass refraction, sharp shadows. Looks like a clean offline raytracer. Real-time on hardware RT.
pathtraceMonte Carlo path. Sample BRDF, spawn next ray, accumulate throughput, Russian-roulette terminate.Many rays, accumulated over framesPhysically correct global illumination. Soft shadows from area lights, indirect bounce, color bleeding. The headline mode.

Switching modes resets the accumulation buffer (since accumulation is meaningless for raytrace and preview). Camera movement still resets accumulation in pathtrace mode as before. Implementation is a single switch on the mode constant inside the kernel — three code paths, one shader, shared intersection.

Renderer interface

class Renderer {
public:
    Renderer(rhi::Device* device, uint32_t window_w, uint32_t window_h);
    void SetScene(const Scene& scene);
    void SetMode(RenderMode mode);                 // pathtrace | raytrace | preview
    void SetResolutionScale(float scale);          // applied to internal render target
    void RenderFrame(const Camera& camera, rhi::TextureHandle output);
    void ResetAccumulation();
    uint32_t GetSampleCount() const;

private:
    rhi::Device* device_;
    uint32_t           window_w_, window_h_;        // swapchain dims
    uint32_t           internal_w_, internal_h_;    // = window × scale, aspect preserved
    RenderMode         mode_ = RenderMode::PathTrace;
    rhi::TextureHandle accum_hdr_;                  // RGBA32F at internal res
    rhi::PipelineHandle render_pipeline_;           // single kernel, mode-switched
    rhi::PipelineHandle tonemap_pipeline_;          // also handles upscale
    rhi::BufferHandle  scene_buffer_;
    rhi::BufferHandle  camera_buffer_;
    uint32_t           sample_count_ = 0;
};

Resolution decoupling — the window stays at 1080p

The render path uses two distinct resolutions:

  • Window resolution: set by app_window_width / app_window_height, default 1920×1080. The swapchain matches this. The window the user sees is always this size.
  • Internal resolution: window × r_resolution_scale, applied to both dimensions. Aspect ratio always preserved. r_resolution_scale 0.5 on a 1920×1080 window gives 960×540 internal. r_resolution_scale ≈ 0.13 gives ~256×144 internal.

Per frame: the render kernel writes to an HDR accumulation texture at internal resolution. The tonemap pass reads the accumulation, applies ACES, gamma-corrects, and upscales to the swapchain via bilinear filtering on the sample. The user sees a 1080p image; the work happened at internal resolution. No letterboxing, no aspect mismatch — both dimensions scaled by the same factor against a window with the same aspect.

This is what makes the software backend usable: it can render at 256×144 internal while still presenting a 1080p window. The image will look low-resolution (because it is), but it won't look broken or stretched.

§ 07

Feature plan

FPS camera

Standard free-fly: WASD + mouse look, Shift to sprint, Space/Ctrl for vertical. Stored as position + yaw + pitch (avoids gimbal issues at the cost of no roll, which we don't need). Pitch clamped to ±85°.

Camera state changes reset path tracer accumulation. The accumulation reset is also the trigger for "stop denoising and re-converge."

Analytic primitives

Three primitive types in the early phases, intersected analytically in the kernel without an acceleration structure:

PrimitiveIntersectionStorage
SphereQuadratic, closed-formcenter, radius, material
PlaneSingle dot productpoint, normal, material
TriangleMöller-Trumborethree vertices, material

Used early for testing the path tracer before mesh loading exists. Stays useful for procedural test scenes after meshes arrive.

Materials

Three BRDFs in v1, stored as a tagged union on the GPU:

  • Lambert — albedo / π, cosine-weighted hemisphere sampling.
  • Metal (conductor) — perfect or glossy reflection via GGX importance sampling. Albedo tints reflectance.
  • Dielectric — Fresnel-weighted reflect/refract using Snell's law and Schlick's approximation. Parameterized by IOR (1.5 = glass, 1.33 = water).

Path tracing loop: bounce up to N times (N=8 in v1), break on miss (return env color or zero) or emissive hit, multiply throughput each bounce, Russian-roulette terminate after depth 4.

§ 08

Mesh CSG via Manifold — the headline

The library that makes this feasible: Manifold, Google's MIT-licensed mesh boolean library. Robust by construction (always produces manifold output from manifold input), exact, fast — single-digit milliseconds for moderate operands. OpenSCAD switched to it. That's the budget that makes interactive editing work.

Strategy: bake on CPU, render via hardware RT on GPU

The CSG layer is a tree on the CPU side:

  • Leaves are imported triangle meshes (glTF/OBJ) with optional transforms.
  • Inner nodes are boolean ops (union / intersect / subtract) over their children.

When the user edits a parameter (slider drags a leaf transform, toggles an op), we walk the dirty path up to the root, recomputing affected subtrees with Manifold. The result is one flat triangle mesh per "CSG body". That mesh becomes a BLAS in the hardware RT pipeline like any other mesh.

namespace csg {

enum class Op { Leaf, Union, Intersect, Subtract };

struct Node {
    Op       op;
    glm::mat4 transform = glm::mat4(1);

    // Leaf-only
    MeshAssetId leaf_mesh;
    uint32_t    leaf_material_id;

    // Inner-node-only
    std::unique_ptr<Node> left;
    std::unique_ptr<Node> right;

    // Cache
    manifold::Manifold cached_result;
    bool dirty = true;
};

class Evaluator {
public:
    BakedMesh Evaluate(Node& root);
    void      MarkDirty(Node& node);
};

struct BakedMesh {
    std::vector<glm::vec3> positions;
    std::vector<glm::vec3> normals;
    std::vector<uint32_t>  indices;
    std::vector<uint32_t>  face_material_ids;
};

}

Materials through CSG

Manifold tracks per-triangle "originalIDs" through boolean ops. We map originalID → material via a side table built at leaf-conversion time. In the hardware-RT closest-hit path, the shader does a single buffer lookup with primitiveID to get the material.

Effect: subtract a metal sphere from a glass box → walls of the resulting hole render as metal, outer surface as glass, no special-casing.

GPU update path on edit

  1. CPU re-evaluates dirty subtree on a worker thread → new BakedMesh.
  2. Upload positions/normals/indices to GPU buffers (reuse if size unchanged).
  3. Rebuild BLAS (full rebuild — topology changed; refit only valid for vertex-position-only changes).
  4. Update the affected TLAS instance entry; rebuild TLAS (cheap).
  5. Reset path-tracer accumulation.
  6. Continue tracing next frame.

For a slider drag on a moderately complex CSG node: ~5–15 ms total. Feels real-time.

Why bake instead of trace-time CSG?

Bake (Manifold)Interval-based
Per-ray costNormal closest-hit (cheap)All-hits enumeration (expensive)
Edit cost5–15 ms (Manifold + BLAS)Zero
Animated booleansPainfulFine
Nested CSGFree at trace timeShader complexity grows with depth
Code complexityModerate (mostly host-side)High (shader interval logic)

We start with bake. If interactive edit lag ever becomes the bottleneck for a specific use case, interval-based CSG can be added as an opt-in mode for the body currently being edited.

Practical caveats

  • Operand meshes must be manifold and watertight. Manifold rejects malformed input. Imported assets often need a repair pass — Blender's "Mesh: Clean Up" operators handle this offline, or Manifold's basic repair.
  • Self-intersecting input meshes break things. Same answer: clean before import.
  • Manifold uses exact arithmetic for predicates that matter. This is why it's the right choice over libigl/CGAL (slower) or Cork (less robust).
  • Run Manifold evaluation on a worker thread. Don't block the render loop on a 50ms worst-case bake.
§ 09

Future: SDF objects

An earlier draft of this design had SDFs (Signed Distance Functions) as a parallel geometry path alongside hardware-RT meshes. They were dropped from v1 scope after weighing the hardware tradeoff: hardware ray tracing accelerates only triangle BVHs. Sphere tracing — the SDF traversal model — is uniform compute work with no fixed-function acceleration on any GPU.

For a Mac M4 Max specifically, the calculus is sharp: the M4's standout hardware feature is its ray tracing cores. A renderer that uses them runs well above what raw FLOPS would predict. A renderer that skips them competes on compute alone, where dedicated GPUs win. Building around SDFs trades away the part of the machine that makes this project interesting.

That said, SDFs remain attractive for things meshes can't easily express: smooth-blended shapes (smoothUnion(d1, d2, k)), procedural surfaces (gyroids, fractals), domain warping, infinite repetition. The architecture leaves the door open to add them as a parallel geometry path in a future version, where every ray would query both the mesh BVH and a list of SDF objects, taking the closest hit. Concretely, the path tracer's intersection function is the only place the change would land — the renderer, scene, and console plumbing don't need to know the difference.

What an SDF addition would look like

  • Primitives: sphere, box, cylinder, torus, plane (analytic SDF formulas, e.g. Inigo Quilez's catalog).
  • Operations: union/intersect/subtract via min/max/max-negate; smooth variants via softmin with parameter k.
  • Storage: small SDF tree per object (primitives + ops + transforms), packed into a structured buffer.
  • Traversal: sphere tracing with central-difference normal estimation.
  • Interaction with meshes: each ray queries mesh TLAS and each SDF object, takes closest hit. The two systems shadow each other and reflect off each other for free.
  • Limitation: cross-system CSG (mesh ⊖ SDF or vice versa) remains out of scope. The two representations don't combine without either meshing the SDF or building a hybrid interval tracer — large work for niche payoff.

Roughly a one-phase addition when prioritized. Not in v1.

§ 10

Memory manager & hardware introspection

See §3 of the principles doc for the rules. Brief summary of the architecture:

Allocator stack

  • Frame arena — bump allocator, triple-buffered, reset at frame end. Per-frame transient data.
  • Pool allocators — fixed-size object pools for jobs, BVH nodes, log entries. Free-list inside, O(1) alloc/free.
  • Persistent heap — mimalloc-backed, default for long-lived heap allocations.
  • GPU heap allocator — wraps Metal MTLHeap / Vulkan VMA. Sub-allocates from large heap blocks to avoid per-allocation API cost.

Tagging

Every allocation tagged with a MemTag (Scene, MeshAssets, Textures, GpuBuffers, Csg, Console, JobSystem, Render, Misc). Per-tag stats: total bytes, peak bytes, alloc count. Each tag has a configurable hard budget (cvar mem_budget_<tag>_mb).

Gathered once at startup, exposed as read-only cvars:

§ 11

Network console

The renderer process is a server. It opens a TCP port (default 27960) speaking WebSocket + JSON, plus a separate line-protocol port (default 27961) for shell scripting. Commands and queries arrive from clients; results and event streams flow back.

Two primitives, both inspired by Quake/Source:

  • CVars — named, typed variables with a value, default, flags (ARCHIVE, READONLY, CHEAT), and an optional OnChange callback. r_max_bounces 8, cam_fov 60.
  • Commands — named functions that take string arguments. screenshot, scene_load mybox.toml, r_backend metal.

Built on top: aliases, keybindings, exec <file> for running .cfg scripts, autoexec.cfg at startup. The full list of cvars and commands lives in the protocol doc.

Frontend

A single index.html + small vanilla JS file embedded in the binary, served over the same port at http://localhost:27960/. No React, no build step, no node. The web UI opens a WebSocket back to the server and renders a console + dashboard. Embed via CMake's file(READ ... HEX) or a small bin2c-style step.

Security

  • Bind to 127.0.0.1 only by default.
  • Origin check WebSocket upgrades — only accept http://localhost:<port> or null (raw WS).
  • Optional token auth (cvar net_require_auth 1); token printed at startup.
  • No HTTPS for localhost — would require self-signed certs and gives nothing useful.
  • For LAN/remote use: SSH tunnel (ssh -L 27960:localhost:27960 box), don't bind to a public interface.

§ 13

Open questions

  • MoltenVK ray tracing extension status. Verify at the start of Phase 4 (Vulkan backend bringup). If incomplete, fallback path: Vulkan backend uses a compute-shader software ray tracer instead of VK_KHR_ray_tracing_pipeline. Slower, but the abstraction stays valid.
  • Slang CPU target maturity. Spike at start of Phase 2 (Software backend) — confirm Slang generates usable C++ for our shader patterns. If not, fallback: write the path tracer twice (once in Slang for GPU, once in C++ for CPU). Significant cost; we want to avoid this.
  • Bindless resources. Likely needed when material count grows past ~16. Add lazily; not a Phase 0 commitment.
  • Mesh repair on import. Manifold rejects non-manifold input. v1 errors out and tells the user to fix in Blender; v2 might integrate a basic repair pass.
  • Animated booleans / interval-based CSG. Not in v1. If interactive edit lag becomes bottlenecked on Manifold for any specific use case, add interval CSG as an opt-in mode.
  • SDF objects. Discussed in §9. Not in v1. Trades away hardware RT advantage; revisit when the v1 feature set is solid and the desire for procedural / smooth-blended geometry is concrete.
  • Cross-system CSG (mesh ⊖ SDF or vice versa). Out of scope even if SDFs are added. Would need either meshing the SDF (marching cubes) or a hybrid interval tracer across both systems.
  • Windows bringup. Phase 12. Same Vulkan backend, native this time. Validates the abstraction against a real driver instead of MoltenVK.
