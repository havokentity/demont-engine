Console Protocol

The wire format and command catalog. Every external tool — the embedded web UI, shell scripts, custom Python automation — speaks one of the protocols defined here. There is no other API.

The renderer process opens two sockets at startup. WebSocket on port 27960 speaks JSON for the rich console UI. Plain TCP on 27961 speaks newline-delimited text for shell pipelines (echo screenshot | nc localhost 27961). Both surfaces dispatch into the same CVar/Command system. Anything you can do from one, you can do from the other.

§ 01

CVars & Commands — the two primitives

Every external interaction reduces to one of two operations: read or write a CVar, or invoke a Command. These two concepts are inherited from the Quake/Source console lineage and they cover every interactive engine surface we need.

CVars

A CVar is a named variable with a string-stored value, a default, optional flags, and an optional change callback. The string storage is deliberate — keeps the protocol uniform — and we coerce on access (GetInt(), GetFloat(), GetBool()).

struct CVar {
    std::string         name;
    std::string         value;
    std::string         default_value;
    std::string         description;
    uint32_t            flags;        // ARCHIVE | READONLY | CHEAT
    std::function<void(const CVar&)> on_change;
};

Flags:

FlagMeaning
ARCHIVEPersisted to config.cfg on shutdown; reloaded on startup. User preferences.
READONLYSet by the engine, not the user. sys_* introspection cvars.
CHEATRequires dev_cheats 1 before mutation is allowed. Debug-only knobs.

Commands

A Command is a named function with a callback that takes a list of string arguments. Argument parsing is the command's responsibility.

struct Command {
    std::string name;
    std::string description;
    std::function<void(std::span<std::string_view>)> callback;
};

Naming convention

Names are snake_case with a namespace prefix matching the subsystem:

PrefixOwner
r_Renderer (bounces, samples, backend, debug visualizations)
sys_Hardware introspection (read-only)
mem_Memory budgets and reports
cam_Camera controls
csg_Mesh CSG editing (Phase 9)
net_Console server config
dev_Developer / debug toggles
app_App-level (window, vsync, target FPS)
CVars and commands are the engine's only API to the outside world. Every tweakable becomes a CVar; every action becomes a Command. There are no other configuration mechanisms.
§ 02

Transport layer

SurfacePortWire formatUse case
WebSocketnet_port (default 27960)JSON messagesWeb UI, rich clients, event subscriptions
HTTP (same port)27960Static filesServes the embedded index.html + JS
Plain TCPnet_line_port (default 27961)Newline-delimited textShell pipelines, nc, simple scripts

Both sockets bind to 127.0.0.1 by default. See §9 Security.

Threading model

Network I/O runs on dedicated threads inside civetweb. Incoming commands queue onto a lock-free SPSC ring; the main thread drains the ring once per frame and dispatches into the Console. Replies and events queue back through a second ring to the network thread. This means console execution always happens on the main thread, in sync with the renderer — a command can mutate any engine state without locking.

§ 03

WebSocket message format

Every message is a JSON object with a type field. Optional id field correlates requests to responses; if the client supplies it, the server echoes it back in the response.

Client → Server

exec — execute a console line (the universal escape hatch; equivalent to typing into the console).

→ client
{
  "type": "exec",
  "id": "req-42",
  "line": "r_max_bounces 12"
}
← server
{
  "type": "result",
  "id": "req-42",
  "ok": true,
  "output": "r_max_bounces: 8 -> 12"
}

get_cvar — read a CVar's current value and metadata.

→ client
{
  "type": "get_cvar",
  "id": "req-43",
  "name": "r_max_bounces"
}
← server
{
  "type": "result",
  "id": "req-43",
  "ok": true,
  "cvar": {
    "name": "r_max_bounces",
    "value": "12",
    "default": "8",
    "flags": ["ARCHIVE"],
    "description": "Maximum path tracer bounce depth"
  }
}

set_cvar — write a CVar. Functionally identical to exec "name value", but explicit.

→ client
{
  "type": "set_cvar",
  "id": "req-44",
  "name": "cam_fov",
  "value": "90"
}
← server
{
  "type": "result",
  "id": "req-44",
  "ok": true
}

list_cvars / list_commands — enumeration. Optional prefix filter for tab completion.

→ client
{
  "type": "list_cvars",
  "id": "req-45",
  "prefix": "r_"
}
← server
{
  "type": "result",
  "id": "req-45",
  "ok": true,
  "cvars": [
    { "name": "r_backend", "value": "metal", ... },
    { "name": "r_max_bounces", "value": "8", ... },
    ...
  ]
}

subscribe / unsubscribe — start or stop receiving events on named topics.

→ client
{
  "type": "subscribe",
  "topics": ["log", "frame_stats", "mem_stats"]
}
← server
{
  "type": "result",
  "ok": true,
  "subscribed": ["log", "frame_stats", "mem_stats"]
}

Server → Client

result — response to a request, correlated by id. Shown above.

event — pushed message on a subscribed topic. No id.

← server (push)
{
  "type": "event",
  "topic": "frame_stats",
  "ts": 1714754321.482,
  "data": {
    "fps": 142.3,
    "frame_ms": 7.03,
    "trace_ms": 5.61,
    "tonemap_ms": 0.42,
    "spp": 487,
    "backend": "metal",
    "resolution": [2560, 1440]
  }
}

Error responses

← server
{
  "type": "result",
  "id": "req-46",
  "ok": false,
  "error": "unknown cvar: r_potato"
}
§ 04

Line protocol (port 27961)

The line protocol is the WebSocket protocol with the JSON peeled away. Every input line is treated as if wrapped in {"type":"exec", "line": "..."}; output is the rendered output string from the result.

Why have it: echo, nc, xargs, shell loops. WebSocket needs a client library; line protocol needs nothing but a TCP socket.

$ nc localhost 27961
sys_info
CPU: Apple M4 Max (12P + 4E)
RAM: 64 GB
GPU: Apple M4 Max (40 cores, 64 GB unified, hardware RT)
OS:  macOS 15.1

r_max_bounces
r_max_bounces = 8

r_max_bounces 16
r_max_bounces: 8 -> 16

screenshot
saved screenshots/2026-05-03_17-23-04.png
^D
$

Errors come back prefixed error: :

$ echo "r_potato" | nc localhost 27961
error: unknown cvar: r_potato

The line protocol does not push events. Subscribe to topics via WebSocket if you want a stream. This is a deliberate limitation — keeps the line port stateless and pipeable.

§ 05

Event stream topics

Topics are subscribed individually. Each emits at its own cadence.

TopicCadencePayload sketch
log On every log call { level, source, message, ts } where level is info|warn|error
frame_stats Once per frame (capped to ~10 Hz over the wire to avoid flooding) { fps, frame_ms, trace_ms, tonemap_ms, spp, backend, resolution }
mem_stats 1 Hz { tags: { Scene: { bytes, peak, count }, Csg: {...}, ... } }
gpu_stats 1 Hz { allocated_bytes, budget_bytes, device_name }
backend_change On Rebuild { from, to, rebuild_ms }
csg_change On CSG re-bake (Phase 9+) { root_node_id, bake_ms, blas_rebuild_ms, tri_count }

§ 08

.cfg scripts & aliases

A .cfg file is plain text, one console line per row, comments with //. exec foo.cfg runs each line through the same parser as a typed command.

autoexec.cfg in the working directory runs at startup automatically (after config.cfg is loaded for ARCHIVE cvars). Use it to script your favorite scene loadout, key bindings, or perf presets.

// autoexec.cfg
r_backend metal
r_max_bounces 12
cam_fov 65

alias quality "r_resolution_scale 1.0; r_max_bounces 16; r_denoiser off"
alias fast    "r_resolution_scale 0.5; r_max_bounces 4;  r_denoiser metalfx"

bind F5 screenshot
bind F9 reload_shaders

scene_load assets/scenes/cornell_drilled.toml
echo "ready"

Aliases are evaluated at invocation time, so they pick up later cvar changes. They can chain commands using ; as a separator.

§ 09

Security

The console exposes total control over the renderer process — backend switches, file I/O via scene_load, screenshot paths. Treat it like an admin shell, because that's what it is.

Defaults that protect you

  • HARD Both sockets bind to 127.0.0.1 only. Other machines on your network cannot reach them. net_bind_address can override; don't, unless you read this whole section.
  • HARD WebSocket upgrades reject any Origin header that isn't http://localhost:<port>, http://127.0.0.1:<port>, or absent (raw clients like wscat). This blocks DNS-rebinding attacks where a malicious website tries to reach your local renderer through a browser tab.
  • HARD No HTTPS, no WSS. Localhost loopback doesn't traverse a network; encryption adds nothing useful and would require self-signed certs that browsers hate.

Optional token auth

Set net_require_auth 1 and the server prints a randomly-generated token at startup:

[net] WebSocket   ws://127.0.0.1:27960/?token=8f3c2e1a0b9d
[net] Line proto  127.0.0.1:27961  (first line: AUTH 8f3c2e1a0b9d)
[net] HTTP UI     http://127.0.0.1:27960/?token=8f3c2e1a0b9d

WS clients must present the token as the ?token= query parameter; line clients must send AUTH <token>\n as their first line. Without auth, the server closes the connection.

You generally don't need this for local-only use — loopback bind + Origin check covers the realistic attack surface. Turn it on when you bind to a non-loopback interface, run untrusted browser extensions, or work in a shared-machine environment.

Remote access — do this, not that

The right way

SSH tunnel:

ssh -L 27960:localhost:27960 -L 27961:localhost:27961 my-mac.local

Then visit http://localhost:27960 on the local machine. The renderer keeps binding to loopback only; SSH handles authentication, encryption, and remote reachability. This is the supported remote workflow.

The wrong way

Setting net_bind_address 0.0.0.0 exposes the console to every device on your LAN. If you ever do this, you must also set net_require_auth 1, and you should still expect that anyone who can reach your machine can crash, reconfigure, or exfiltrate from your renderer. The convenience cvar net_lan_mode 1 does both at once with a warning. Use only on isolated dev networks.

Startup banner

The server prints connection info every startup. The agent should make sure this is unmissable in the log.

Connection examples

Embedded web UI

Open http://localhost:27960/ while the renderer is running. The page loads the embedded JS, opens a WebSocket to the same origin, subscribes to log + frame_stats + mem_stats, and presents a console + dashboard. No build step, no install — vanilla JS straight out of the binary.

Python WebSocket client

import asyncio, json, websockets

async def main():
    async with websockets.connect("ws://localhost:27960/") as ws:
        # one-shot command
        await ws.send(json.dumps({"type": "exec", "id": "1",
                                  "line": "r_max_bounces 12"}))
        print(await ws.recv())

        # subscribe + stream stats
        await ws.send(json.dumps({"type": "subscribe",
                                  "topics": ["frame_stats"]}))
        async for msg in ws:
            evt = json.loads(msg)
            if evt.get("topic") == "frame_stats":
                print(f"{evt['data']['fps']:.1f} fps")

asyncio.run(main())

Shell loop driving an animation

for i in $(seq 0 360); do
  echo "csg_set_transform 5 0 0 0 0 $i 0 1 1 1" | nc localhost 27961
  sleep 0.016
done

(That's the "we don't need an embedded scripting language" argument made concrete. Animations are a shell loop. Procedural scene generation is a Python script. Hot reload of materials is a file watcher piping into nc. The console is the API.)
