Modern C++
Cheatsheet

A reference for experienced C/C++ engineers reconnecting with the modern dialect. Covers the 14 features you'll see across this codebase, with examples that focus on what's genuinely new versus what just looks new.

§ 00

Preface — who this is for

If you wrote C++ in the Quake III / Doom 3 era and have been away since, modern C++ looks like a different language. It mostly isn't. The instincts that mattered then — memory ownership, cache behavior, what's actually happening at the assembly level — still matter, and they're still the hard part.

What changed is mostly syntax sugar (lambdas, range-based for, auto) and one genuinely new idea (move semantics). Everything else is library additions you can adopt as you encounter them.

This doc walks through the features used across this project, marked by what they actually are:

  • REALLY NEW — actual new ideas, not just syntax. Update your mental model.
  • SYNTAX — same idea you've always done, expressed cleaner.
  • SUBTLE — looks innocent, has a sharp edge worth knowing about.

This is a reference, not a tutorial. Skim the section that matches the syntax you just saw in the code; come back when something else looks unfamiliar.

§ 01

The mental model shift, in one paragraph

The single biggest change between C++03 and modern C++ is move semantics. Pre-C++11, every assignment and return was a copy unless you took a pointer or reference. Post-C++11, the language understands "transfer ownership" as a first-class operation, and the standard library uses it everywhere. This is what makes std::vector<Mesh> as a return type efficient, what makes unique_ptr work, and what makes RAII finally nice instead of just correct.

Everything else — lambdas, auto, range-based for, structured bindings — is convenience that you'll absorb in a week of actual coding. Move semantics is the one piece that needs deliberate attention. We'll cover it in §4.

If you only internalize one new thing from modern C++, make it move semantics. The rest is gravy.
02 auto and structured bindings C++11 / 17 · SYNTAX

auto means "compiler, figure it out." Structured bindings let you destructure into named variables.

Pre-modern
std::map<std::string, int>::iterator
    it = scores.find("alice");

for (size_t i = 0;
     i < pairs.size(); ++i) {
    int   key   = pairs[i].first;
    float value = pairs[i].second;
    use(key, value);
}
Modern
auto it = scores.find("alice");

for (auto& [key, value] : pairs) {
    use(key, value);
}

Rules of thumb

  • Use auto when the type is obvious from the right side, or when the type doesn't matter (iterators, lambdas).
  • Spell out the type when it documents intent or catches errors. float not auto when the right side is 1.0.
  • auto& for references, const auto& for read-only refs, auto* for pointers (mostly cosmetic but reads better).
Pitfalls
  • auto x = vec[0]; copies. Use auto& x = vec[0]; if you don't want a copy.
  • auto on integer literals gives you int, even where you'd expect size_t. Prefer size_t spelled out for indices.
03 std::unique_ptr and RAII C++11 · REALLY NEW

Heap allocation that frees itself when it goes out of scope. The single best feature of modern C++.

Pre-modern
Renderer* r = new Renderer(device);
// ... 200 lines later ...
if (error_path)
    return;     // leaked
delete r;
Modern
auto r = std::make_unique<Renderer>(device);
// ...
if (error_path)
    return;     // r is destroyed cleanly
// also destroyed at end of scope

How it actually works

A unique_ptr<T> is a struct that holds a T*. Its destructor calls delete on that pointer. It's non-copyable (you can't have two of them owning the same thing) but movable (you can transfer ownership). Sizeof is the same as a raw pointer.

Passing around

The thing you'll do almost always:

// Take by reference if you don't transfer ownership
void RenderFrame(Renderer& r);

// Take by unique_ptr-by-value (with std::move at the call site)
//   if you DO transfer ownership
void StoreRenderer(std::unique_ptr<Renderer> r);

// Caller:
auto r = std::make_unique<Renderer>(device);
RenderFrame(*r);                  // by reference
StoreRenderer(std::move(r));      // hand off ownership
// after move, r is empty (nullptr)
Pitfalls
  • Never write new in modern C++ outside of make_unique's implementation. If you find yourself reaching for new, you've forgotten the rules.
  • Don't pass const unique_ptr<T>& when you mean const T&. The unique_ptr-ness is irrelevant to the callee; pass the underlying reference.
  • Don't std::move a unique_ptr on the way out of a function (return std::move(p);). Just return p; — the compiler does the right thing, and explicit move can defeat NRVO.

The rest of the smart pointer family

Modern C++ has three smart pointers in the standard library, plus weak_ptr as a companion to shared_ptr. They are not interchangeable. This project's policy on each:

TypePolicyCost
std::unique_ptr<T> Default for owned heap allocations. Use freely. Zero — same size as T*, no atomic ops, no separate allocation.
std::shared_ptr<T> Requires justification. Default is no. Atomic inc/dec on copy/destroy. Separate control block (unless make_shared). Cycle leaks if you're not careful.
std::weak_ptr<T> Requires justification. Existence usually indicates a design problem. Cheap on its own; only exists alongside shared_ptr.
Raw T* Fine for non-owning references. Modern C++ didn't kill raw pointers; it killed raw owning pointers. Zero.
T& reference Fine when null doesn't make sense. Function parameters mostly. Zero.

Why shared_ptr is policy-restricted

Three independent reasons, and they compound:

  • Atomic refcount cost. Every copy of a shared_ptr does an atomic increment. Every destruction does an atomic decrement. Atomics aren't free — they prevent compiler reordering and force memory-barrier-style synchronization. On contended atomics across cores, this gets expensive faster than people expect.
  • Cycles leak silently. If A shared_ptrs B and B shared_ptrs A, neither's count ever hits zero. You need a weak_ptr to break the cycle. This is a real bug class, not a theoretical one — it's how observer patterns and parent/child graphs leak.
  • It's usually a design smell. Reaching for shared_ptr as a default often means ownership wasn't thought through. "Eh, multiple things might own this" is the smell. A well-designed system has clear ownership: one place owns each object, others reference it non-owningly. unique_ptr + raw pointers/references express this. shared_ptr obscures it.

There are real cases where shared_ptr is right — caches with unpredictable consumer lifetimes, async tasks that need to keep their inputs alive past the originator's scope, certain graph structures with no clear owning root. But those should be deliberate decisions with a comment explaining the choice, not the default.

What this project actually uses

Roughly, the distribution you'll see across the codebase:

PatternUsed for
std::unique_ptr<T>Engine owning Device, Renderer, Console, JobSystem. CSG tree nodes. Anything else with a clear single owner.
Raw T* (non-owning)RHI hands CommandBuffer* back from AcquireCommandBuffer — the device owns it, you borrow.
T& referenceFunction parameters where null doesn't make sense. RenderFrame(const Camera& cam, ...).
Integer handlesRHI resources (BufferHandle, TextureHandle). Better than pointers for opaque IDs that cross backend boundaries.
std::shared_ptr<T>Likely zero places. If one shows up, it'll be deliberate and commented.
std::weak_ptr<T>Zero places.

The mental shift from pre-C++11: when you see new Foo, your old instinct was "where's the matching delete." The new instinct is "that new shouldn't exist; this should be make_unique<Foo>." References for parameters, raw pointers for non-owning references, smart pointers for ownership, integer handles for opaque resources — and the language stops being a memory management language at all.

04 Move semantics C++11 · REALLY NEW

"I'm done with this object — you can steal its guts." Cheap ownership transfer instead of a deep copy.

This is the one C++ idea you didn't have in 2003 and need to actually internalize. It's how vector returns are efficient, how unique_ptr hands off, how containers grow without quadratic copying.

std::vector<Mesh> source = LoadMeshes();          // owns its contents
std::vector<Mesh> dest   = std::move(source);    // ownership transferred
// no copy happened — just pointer ownership swap inside the vector
// source is now in a valid-but-unspecified state (typically empty)
// source's destructor will still run — moving doesn't change lifetime

What std::move actually is

It's a cast, not a function that does anything. It says "treat this lvalue as an rvalue" — i.e., "I'm done with this, you can steal from it." The actual stealing happens in the move constructor or move assignment operator of the receiving type.

The 90% rule

  • Return values by value. Compiler does the right thing (NRVO + move).
  • Take parameters by const reference for read-only, by value when you'll store a copy.
  • Use std::move explicitly only when transferring ownership (handing a unique_ptr off, putting a vector into a member variable in a constructor).
  • Don't std::move on return. return std::move(x); often prevents compiler optimization. Just return x;.
// Constructor taking a vector and storing it
class Mesh {
    std::vector<Vertex> vertices_;
public:
    explicit Mesh(std::vector<Vertex> v) : vertices_(std::move(v)) {}
    // Take by value, move into member.
    // Caller: Mesh m(LoadVerts());        — efficient, one move
    // Caller: Mesh m(my_verts);           — one copy + one move (acceptable)
};
Pitfalls
  • "Moved-from" objects are still alive. Their destructor runs. They're "valid but unspecified" — usually means empty for STL types.
  • Don't use a moved-from object except to assign to it or destroy it.
  • std::move on a primitive (int, float, raw pointer) does nothing — those types just copy. No harm done, but no benefit either.
05 Range-based for C++11 · SYNTAX

Iterate without indices or iterators. The default loop in modern code.

for (auto& mesh : meshes)        { mesh.transform *= 2.0f; }   // mutate
for (const auto& mesh : meshes)  { Process(mesh); }            // read-only
for (auto mesh : meshes)         { /* COPIED — usually wrong */ }

// With index when needed (C++20 init):
for (size_t i = 0; auto& mesh : meshes) {
    Process(i, mesh);
    ++i;
}

Works on any container with begin()/end(), including C arrays and your own types if you provide them.

Pitfalls
  • The auto mesh form (no &) silently copies every element. For trivial types fine; for anything with a non-trivial copy, costly.
  • Modifying the container inside the loop (push_back, erase) invalidates iterators. Same as always. Take an indexed loop or copy the container if you need that.
06 Lambdas C++11 · SYNTAX

Anonymous functions you can declare inline. Replaces functor structs and most callback boilerplate.

// Syntax: [capture](args) -> ret { body }
// Return type usually deduced; specify when needed.

auto square = [](int x) { return x * x; };
int y = square(5);  // 25

// Capture by value, by reference, or specific things
int  base = 10;
auto add_base    = [base](int x)    { return x + base; };       // copies base
auto inc_base    = [&base](int x)   { base += x; };             // captures ref
auto with_both   = [&base, mul=2](int x) { return base * mul + x; };

// Use in algorithms
std::sort(meshes.begin(), meshes.end(),
    [](const Mesh& a, const Mesh& b) { return a.tri_count < b.tri_count; });

Captures

  • [] — capture nothing
  • [x] — capture x by value
  • [&x] — capture x by reference
  • [=] — capture everything used by value (avoid; explicit is better)
  • [&] — capture everything used by reference (acceptable in tight scope)
  • [name = expression] — init capture (C++14), useful for moving things in
Pitfalls
  • Capturing by reference and outliving the captured. A lambda that captures a stack variable by reference and is stored in a vector that survives the function: dangling reference, undefined behavior.
  • Don't pass lambdas as std::function in hot paths. Type-erasure heap-allocates and prevents inlining. For algorithm callbacks, pass them as templates (std::sort already does this).
07 std::expected<T, E> C++23 · REALLY NEW

A return type that's either a T or an E. Errors without exceptions.

This is Rust's Result finally landing in C++. For us, it's the standard error-handling mechanism — exceptions are forbidden (see principles §8), and bool-out-pointer is ugly.

Pre-modern
bool LoadMesh(const char* path,
              Mesh* out_mesh,
              char* out_err, size_t err_len);
// caller manages buffers,
// checks return, juggles strings
Modern
std::expected<Mesh, std::string>
LoadMesh(std::string_view path);

auto result = LoadMesh("bunny.glb");
if (!result) {
    LOG_ERROR("{}", result.error());
    return;
}
Mesh m = std::move(*result);

Common patterns

// Constructing
std::expected<Mesh, std::string> LoadMesh(std::string_view path) {
    auto file = OpenFile(path);
    if (!file)  return std::unexpected(fmt::format("can't open {}", path));

    auto bytes = ReadAll(*file);
    if (!bytes) return std::unexpected("read failed");

    return ParseMesh(*bytes);  // Mesh implicitly wraps as expected
}

// Consuming
auto result = LoadMesh(path);
if (!result) {
    return std::unexpected(result.error());  // forward the error
}
auto& mesh = *result;  // or: Mesh m = std::move(*result);

// Boolean check, dereference, .error() — that's the whole API for daily use.
Pitfalls
  • Don't ignore the return value. [[nodiscard]] on the function makes the compiler warn; we mark all expected-returning functions with it.
  • Don't dereference an expected without checking. Same shape as null-pointer-deref bugs in C.
08 std::span & std::string_view C++17 / 20 · REALLY NEW

Non-owning views over contiguous data. Replaces every (T*, size_t) pair you've ever passed.

// Pre-modern: pointer + length, easy to mis-pair
void ProcessVertices(const Vertex* verts, size_t count);

// Modern: span carries both
void ProcessVertices(std::span<const Vertex> verts);

// Caller — span constructs from anything contiguous
std::vector<Vertex> v = ...;
ProcessVertices(v);                     // implicit conversion
ProcessVertices({v.data(), 100});       // first 100 only
ProcessVertices(v.subspan(10, 50));     // sub-range

// In the function:
for (const auto& vert : verts) { ... }  // works
verts.size();                           // 100
verts[3];                               // bounds-checked in debug

string_view is the same idea for strings: a (const char*, size_t) pair that doesn't own its memory. Use as a function parameter when you don't need the callee to own the string.

// Old:    void Log(const char* msg);  or  void Log(const std::string& msg);
// Modern: void Log(std::string_view msg);
//   accepts both C strings and std::string without conversion
Pitfalls
  • Don't store a span/string_view as a member of a long-lived object. The underlying memory might go away. They're parameter types, not storage types.
  • string_view is not null-terminated. Don't pass .data() to C APIs expecting null-termination.
09 Designated initializers C++20 · SYNTAX

Initialize struct fields by name at the call site. Self-documenting.

Pre-modern
TextureDesc desc(2048, 2048,
    Format::RGBA8,
    Usage::Storage, 1);
// what was the 4th arg again?
Modern
TextureDesc desc{
    .width  = 2048,
    .height = 2048,
    .format = Format::RGBA8,
    .usage  = Usage::Storage,
    .mips   = 1,
};

Use it for any aggregate (struct without user-defined constructors) with more than two fields. Fields must be in declaration order (unlike C, where any order is allowed).

10 enum class C++11 · SYNTAX

Scoped, type-safe enums. Use these always, never plain enum.

Pre-modern
enum BackendType {
    Software, Metal, Vulkan
};
// pollutes namespace; implicit
// converts to int; collisions
// with other enums
BackendType b = Software;
int x = b;        // works, ugh
if (b == 0) ...   // works, ugh
Modern
enum class BackendType {
    Software, Metal, Vulkan
};
BackendType b = BackendType::Software;
// int x = b;     // compile error
// b == 0         // compile error
if (b == BackendType::Software) ...

// Specify underlying type if needed:
enum class Flags : uint32_t {
    None = 0, Archive = 1, ReadOnly = 2,
};

If you actually want the integer conversion (e.g., using as an array index), it's an explicit cast: static_cast<size_t>(BackendType::Metal). Slightly more typing, much fewer bugs.

11 Strong-typed IDs Idiom · SYNTAX

Wrap integer IDs in single-member structs to make them distinct types. Catches whole bug classes for free.

Loose
using MeshId     = uint32_t;
using MaterialId = uint32_t;

void Render(MeshId m, MaterialId mat);

MeshId mesh = 5;
MaterialId mat = 7;
Render(mat, mesh);  // compiles, wrong
Strong
struct MeshId     { uint32_t v; };
struct MaterialId { uint32_t v; };

void Render(MeshId m, MaterialId mat);

MeshId mesh{5};
MaterialId mat{7};
Render(mat, mesh);  // COMPILE ERROR
Render(mesh, mat);  // OK

Zero runtime cost — sizeof(MeshId) == sizeof(uint32_t). The compiler optimizes away the wrapper. We use this everywhere we have integer handles: MeshId, MaterialId, all the RHI handles (BufferHandle, TextureHandle, PipelineHandle).

Add helpers as needed (comparison, hashing for use as map keys), but keep the type itself tiny.

12 if constexpr C++17 · SYNTAX

Compile-time branching inside templates. Replaces 80% of cases that needed SFINAE in the bad old days.

template<typename T>
void Serialize(T value) {
    if constexpr (std::is_integral_v<T>) {
        WriteInt(value);
    } else if constexpr (std::is_floating_point_v<T>) {
        WriteFloat(value);
    } else {
        WriteBytes(&value, sizeof(value));
    }
}

The branch not taken doesn't have to compile. So this works even when, say, WriteInt(value) wouldn't make sense for a float — that branch is removed at compile time.

Use it for the simple cases where you'd previously write three function overloads with std::enable_if. For anything more complex than this, the principles doc forbids template metaprogramming — write three named functions instead.

13 fmt::print & std::format C++20 / lib · SYNTAX

Python-style format strings, type-safe. printf's ergonomics with iostream's safety, finally.

fmt::print("hello {}\n", name);
fmt::print("frame {}: {:.2f} ms\n", frame_idx, frame_time_ms);
fmt::print("backend: {}\n", backend);  // works for any type with operator<<

// Format to a string instead of printing
std::string s = fmt::format("{} = {}", key, value);

// Width, precision, alignment
fmt::print("{:>10}\n", "right");      // right-aligned in 10 chars
fmt::print("{:08x}\n", 255);          // hex, zero-padded to 8
fmt::print("{:.3f}\n", 3.14159);      // 3 decimal places

We use fmt library rather than std::format directly because compiler support for std::format is still uneven on Apple Clang. Same syntax; switch to standard later if it stabilizes.

Wrapped in our log macros: LOG_INFO("loaded {} meshes", count);. No more printf format-string mismatches at runtime.

14 std::optional<T> C++17 · SYNTAX

A value that might or might not exist. Self-documenting nullability without using a sentinel pointer.

std::optional<CVar> FindCVar(std::string_view name);

if (auto cvar = FindCVar("r_max_bounces")) {
    LOG_INFO("found: {}", cvar->value);
} else {
    LOG_WARN("not found");
}

// Default if missing
auto cvar = FindCVar("r_max_bounces").value_or(default_cvar);

When to use optional vs expected:

  • optional<T> — failure is normal, no error info needed. "Is this thing here? Yes/no."
  • expected<T, E> — failure has reasons that matter. File didn't load, parse failed, etc.
Pitfalls
  • Don't use optional as a "lazy initialize" hack. If something is constructed lazily, say so explicitly.
  • *opt on an empty optional is undefined behavior. Check first or use .value_or().
15 What we skip — · CONTEXT

A short list of celebrated modern C++ features that this project deliberately doesn't use, and why.

FeatureWhy we skip
concepts / requiresCool but rarely earn their cost in practice. If you find yourself writing one, you're probably overengineering the template.
Ranges (std::ranges, views)Compile-time cost is real; readability gains over plain loops are debatable. Plain for loops win.
Coroutines (co_await)Designed for async I/O. We don't have async I/O. Job system handles parallelism.
ModulesCompiler and build-system support still patchy in 2026. Headers and .cpp files for now.
Templated lambdas ([]<typename T>)Niche; readable equivalents exist. Pulls toward template metaprogramming, which is forbidden.
std::variantUseful in theory but visitor pattern with std::visit + lambdas is awkward. Tagged unions written by hand are clearer for our cases (Material, AST nodes).
std::anyType-erased storage with heap allocation. Solves problems we don't have.
ExceptionsForbidden by build flag -fno-exceptions. std::expected handles errors.
RTTIForbidden by build flag -fno-rtti. Tagged unions or type IDs handle runtime type queries.

This is a deliberate aesthetic. Modern C++ has too many features for any one project to use them all coherently; we pick a subset and stay there. See principles §8 for the full enforcement list.
